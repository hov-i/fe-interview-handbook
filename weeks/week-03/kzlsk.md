## 함수 선언문과 함수 표현식은 어떤 차이가 있나요?

- **카테고리**: JavaScript

### 답변

- 함수 선언문과 함수 표현식의 가장 큰 차이는 호이스팅 방식입니다.
- 함수 선언문은 코드 실행 전 함수 전체가 메모리에 등록되기 때문에, 선언 전에 호출이 가능합니다.
- 함수 표현식은 변수 선언부만 호이스팅되고, 함수 할당은 실제 코드 실행 시점에 이루어지므로, 할당 전에 호출하면 에러(let, const)가 발생하거나 undefined(var)가 반환됩니다.
- 따라서 함수 선언문은 코드 순서에 관계없이 자유롭게 호출할 수 있지만, 함수 표현식은 반드시 할당 이후에만 호출해야 합니다.

- 즉시 실행 함수에 대해 알고 있나요?
- 조건문 안에서 함수를 선언할 때 함수 선언문 대신 함수 표현식을 쓰는 게 권장되는 이유는 무엇인가요?

## 이벤트 루프와 태스크 큐에 대해서 알고 있나요?

- **카테고리**: JavaScript

### 답변

- 이벤트 루프는 자바스크립트의 싱글 스레드 특성을 유지하면서 비동기 작업을 처리할 수 있게 해주는 메커니즘입니다.
- 콜 스택이 비어있을 때, 태스크 큐에 대기 중인 콜백을 하나씩 콜 스택으로 옮겨 실행하는 역할을 합니다.
- setTimeout, setInterval, DOM 이벤트 콜백 등이 태스크 큐에 들어갑니다.
- 이 과정을 통해 오래 걸리는 비동기 작업이 메인 스레드를 블로킹하지 않고, 완료된 시점에 콜백만 큐에 등록되어 순차적으로 처리됩니다.

## 이벤트 전파와 위임에 대해서 설명해주세요.

- **카테고리**: JavaScript

### 답변

- 이벤트 전파는 이벤트가 DOM 트리를 따라 전달되는 과정이고, 이벤트 위임은 이 전파 특성을 활용해 상위 요소에서 하위 요소의 이벤트를 처리하는 패턴입니다.
- 이벤트 전파는 캡처링(상위 > 하위) -> 타겟 -> 버블링(하위 > 상위) 3단계로 이루어집니다.
- 이벤트 위임은 자식 요소 각각에 이벤트 리스너를 붙이는 대신, 부모 요소 하나에만 리스너를 등록하고 event.target으로 실제 클릭된 요소를 판별하는 방식입니다.
- 이를 통해서 동적으로 추가/삭제되는 요소에도 별도 리스너 등록 없이 대응할 수 있고, 리스너 개수를 줄여 메모리와 성능 면에서 이점을 얻을 수 있습니다.

## REST API가 무엇인가요?

- **카테고리**: JavaScript

### 답변

- REST는 자원을 URI로 표현하고, 자원에 대한 행위를 HTTP 메서드로 표현하는 아키텍처 스타일입니다.
- 이 원칙을 HTTP 기반으로 구현한 서비스 API를 REST API라고 합니다.
- 대표적인 원칙으로는 URI 식별, 메서드를 통한 행위 표현(GET, POST, PUT, DELETE), stateless 등이 있습니다.
- 이 원칙들을 성실히 지킨 서비스 디자인을 'RESTful' 하다고 표현합니다.
- stateless : 서버가 클라이언트 상태를 저장하지 않고 요청에 필요한 정보를 요청 자체에 담는 것

## React에서 state의 불변성을 유지하라는 말이 있는데 이에 대해서 설명해주세요.

- **카테고리**: React

### 답변

- React는 state의 참조값이 바뀌었는지를 기준으로 변경 여부를 판단하기 때문에, 기존 state를 직접 수정하지 않고 새로운 객체/배열을 만들어 교체해야 합니다.
- 그래서 스프레드 연산자나 배열 메서드로 새로운 참조를 가진 객체/배열을 만들어 state에 전달해야 합니다.
- 불변성을 지키면 변경 감지가 정확해질 뿐만 아니라, 최적화도 신뢰성 있게 동작합니다.

## 언제 useState 대신 useRef를 쓸 수 있나요?

- **카테고리**: React

### 답변

- useState를 대신해서 useRef를 사용할 때는 값이 변경되어도 리렌더링을 유발할 필요가 없거나, DOM 요소에 직접 접근을 해야 할 때 사용할 수 있습니다.
- useState는 값이 바뀌면 리렌더링이 발생하지만, useRef는 .current 값을 변경해도 리렌더링이 발생하지 않습니다.
- 그래서 타이머 ID, 이전 값 저장, 렌더링 횟수 카운트 등과 같이 렌더링과 무관하게 값만 기억하고 싶은 경우에는 useRef를 사용하는 것이 적합합니다.

## 주소창에 google.com을 입력하는 어떤 일이 일어나나요?

- **카테고리**: Browser

### 답변

- URL 해석 > DNS 조회 > TCP/TLS 연결 > HTTP 요청/응답 > 브라우저 렌더링 순서로 진행됩니다.
- 브라우저는 입력 값이 URL인지 검색어인지 판단하고, DNS를 통해 도메인에 해당하는 IP 주소를 조회합니다.
- 조회된 IP로 서버와 TCP 3-way handshake, 이어서 HTTPS라면 TLS handshake를 통해 연결을 맺고, HTTP 요청을 보내 응답을 받습니다.
- 그 후 받은 HTML과 CSS를 파싱해 DOM/CSSOM을 만들고, 이를 결합한 렌더 트리를 기반으로 레이아웃과 페인트 과정을 거쳐 화면에 실제로 그려냅니다.

## 쿠키와 세션에 대해서 설명해주세요.

- **카테고리**: Network

### 답변

- 쿠키와 세션은 둘 다 상태가 없는(stateless) HTTP 특성을 보완하기 위해 사용자 상태를 유지하는 방법이지만, 데이터가 저장되는 위치가 다릅니다.
- 쿠키는 클라이언트에 key-value 형태로 저장되며, 서버가 응답 헤더(set-Cookie)로 내려주면 이후 요청마다 자동으로 함께 전송됩니다.
- 세션은 서버 측에 사용자 정보를 저장하고, 클라이언트에는 세션을 식별할 수 있는 세션 ID만 쿠키 형태로 저장해 매 요청마다 이 ID로 서버가 사용자를 식별합니다.
- 즉, 쿠키는 클라이언트 저장 방식이라 노출/변조 위험이 있고, 세션은 민감한 데이터를 서버에 두기 때문에 상대적으로 안전하지만 서버 자원을 사용한다는 트레이드오프가 있습니다.
