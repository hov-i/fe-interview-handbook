# N주차 - 이름

## var, let, const의 호이스팅 동작 차이
- **카테고리**: JavaScript

### 답변
- var은 호이스팅 되면서 undefined로 초기화 > 선언 전 접근해도 에러 없이 undefined 반환
- let/const는 호이스팅은 되지만 TDZ가 존재하기 때문에 초기화가 안됨 > 선언 전 접근 시 에러 발생

---

## React Hooks에서 클로저가 하는 역할
- **카테고리**: React

### 답변
- useState나 useEffect 등은 클로저로 동작
- 컴포넌트가 리렌더링될 때마다 새로운 렌더링 환경이 만들어지고, 그 시점의 state/props 값을 클로저가 기억
- useEffect 안에서 참조하는 state가 그 렌더링 시점의 값으로 고정되는 stale closure 문제가 생길 수 있음

---

## 렉시컬 스코프와 동적 스코프의 차이
- **카테고리**: JavaScript

### 답변
- 렉시컬 스코프는 함수를 어디서 선언했는지에 따라 상위 스코프가 결정됨
- 동적 스코프는 함수를 어디서 호출했는지에 따라 상위 스코프가 결정됨

## 이벤트 리스너 콜백에서 this가 가리키는 것은?
- **카테고리**: JavaScript

### 답변
- 일반 함수로 작성하면 이벤트가 발생한 DOM 요소를 가리킴
- 화살표 함수로 작성 시 렉시컬 this가 적용돼 상위 스코프의 this를 가리킴

---

## 리액트에서 이벤트 핸들러를 화살표 함수로 쓰는 이유는?
- **카테고리**: React

### 답변
- 클래스 컴포넌트에서 일반 함수는 호출 시 this가 undefined가 되기 때문에 화살표 함수로 선언하거나 bind로 묶어야 함

---

## 클로저와 실행 컨텍스트는 어떤 관계인가요?
- **카테고리**: JavaScript

### 답변

### 답변

- async/await은 promise를 동기 코드처럼 작성 가능하게 해주는 문법으로 가독성이 더 좋음

## 가상 DOM이 실제로 성능상 항상 빠른가요?
- **카테고리**: React

### 답변
- 항상 빠른 건 아님
- 단순한 페이지인 경우 오히려 오버헤드
- 상태 변경이 잦고 DOM 업데이트가 많은 복잡한 UI에서 효과적

---

## key prop이 반영(reconciliation)에서 왜 중요한가
- **카테고리**: React

### 답변
- key를 고유 값으로 주면 어떤 요소가 추가/삭제/이동됐는지 정확히 식별 가능하기 때문에 효율적으로 이전 트리와 비교 가능
- key가 없으면 react는 배열 순서로 엘리먼트를 비교해 불필요한 리렌더링이나 잘못된 DOM 재사용 발생 가능

---

## 마운트와 언마운트가 각각 어떤 시점인가요?
- **카테고리**: React

### 답변

- 언마운트는 컴포넌트가 DOM에서 제거되는 시점으로 페이지 이동, 조건부 렌더링으로 컴포넌트가 사라질 때 해당

## async/await은 Promise와 어떤 관계인가요?
- **카테고리**: CS

### 답변
- async/await은 Promise 위에 얹힌 문법적 설탕
- await은 Promise가 settled 될 때까지 함수 실행을 일시 중단, 내부적으로는 동일하게 마이크로태스크 큐 사용
